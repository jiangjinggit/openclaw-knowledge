# 🧠 MEMORY.md - 长期记忆模板

> 跨会话持久化的记忆，记录经验教训，让 AI 越来越聪明

---

## 📖 使用说明

### 什么是 MEMORY.md？

MEMORY.md 用于记录跨会话的长期记忆，包括：
- ❌ 每次犯错的教训
- ✅ 每次学习的经验
- 🎯 每次重要的决策

### 为什么需要 MEMORY.md？

**没有 MEMORY.md：**
- AI 会重复犯错
- 每次都要重新解释
- 无法积累经验

**有了 MEMORY.md：**
- AI 自动避免重复错误
- 自动应用历史经验
- 持续进化，越来越聪明

---

## 📝 记录格式

### 标准格式

```markdown
### [日期] [类型] [标题]
- **问题**：[发生了什么]
- **根因**：[为什么会发生]
- **修正**：[怎么解决的]
- **预防**：[下次怎么避免]
```

### 类型说明

| 类型 | 说明 | 使用场景 |
|------|------|----------|
| `ERROR` | 错误/失败 | 犯错时记录 |
| `LEARN` | 学习/经验 | 学到新东西时记录 |
| `DECISION` | 重要决策 | 做重大决策时记录 |

---

## 📚 记忆分类

### 1. 错误类（ERROR）

记录失败和教训，避免重复犯错。

**示例1：任务没有验收标准**
```markdown
### 2026-03-06 ERROR 任务没有验收标准
- **问题**：Build 完成任务后，我觉得不满意，但说不出哪里不满意
- **根因**：任务卡没有明确的验收标准
- **修正**：重新定义任务，增加验收标准："3000字、3个真实案例、通过门禁检查"
- **预防**：以后每个任务都必须有可量化、可验证的验收标准
```

**示例2：数据丢失**
```markdown
### 2026-03-06 ERROR 数据丢失
- **问题**：删除文件时没有备份，数据丢失
- **根因**：没有养成备份习惯
- **修正**：从备份恢复（幸好有自动备份）
- **预防**：删除前必须先备份；设置自动备份
```

---

### 2. 学习类（LEARN）

记录学到的经验和最佳实践。

**示例1：配置文件要简洁**
```markdown
### 2026-03-06 LEARN 配置文件要简洁
- **问题**：第一版 SOUL.md 写了 50 条规则，AI 执行混乱
- **根因**：规则太多，AI 无法判断优先级
- **修正**：简化为 10 条核心规则
- **预防**：从简单开始，逐步优化；每次只加 1-2 条规则
```

**示例2：标题公式**
```markdown
### 2026-03-06 LEARN 标题公式
- **问题**：文章阅读量低
- **学习**：好标题 = 趋势词 + 冲突/对比 + 数据锚点
- **应用**：下次写标题时使用这个公式
- **结果**：阅读量提升 3 倍
```

---

### 3. 决策类（DECISION）

记录重要决策的原因和结果，方便复盘。

**示例1：选择 OpenClaw**
```markdown
### 2026-03-06 DECISION 选择 OpenClaw
- **问题**：需要一个 AI 协作系统
- **选择**：OpenClaw（而不是 ChatGPT、Claude、Cursor）
- **原因**：
  1. 主动执行（定时任务）
  2. 团队协作（多 Agent）
  3. 完全本地（数据安全）
- **结果**：效率提升 8 倍
```

**示例2：选择技术栈**
```markdown
### 2026-03-06 DECISION 选择技术栈
- **问题**：需要选择前端框架
- **选择**：React（而不是 Vue、Angular）
- **原因**：
  1. 生态最好
  2. 招人容易
  3. 我最熟悉
- **结果**：开发速度快，没有踩坑
```

- ❌ 沟通误解

- ✅ 总结新经验

### 3. 做重大决策时（必须记录）
- 🎯 选择技术栈
- 🎯 选择工具
- 🎯 选择方向
- 🎯 选择合作伙伴

## 📋 记录原则

### 1. 简洁
一条记录不超过 100 字。

**示例：**
```
❌ 错误：今天我在做任务的时候发现了一个问题，就是任务完成后我觉得不满意，
         但是我又说不出哪里不满意，后来我发现是因为任务卡没有明确的验收标准...

✅ 正确：任务没有验收标准 → 不知道怎么算完成 → 增加验收标准 → 以后必须有验收标准
```

### 2. 具体
不要写"注意细节"，要写"检查验收标准"。

**示例：**
```
❌ 错误：预防：下次小心
✅ 正确：预防：删除前必须先备份；设置自动备份
```

### 3. 可执行
预防措施要可执行，不要写空话。

**示例：**
```
❌ 错误：预防：提高质量意识
✅ 正确：预防：发布前运行门禁检查脚本
```

## 🔍 什么时候查阅？

### 1. 遇到类似问题时
先查 MEMORY.md，看有没有记录。

### 2. 开始新任务前
看看有没有相关经验。

### 3. 定期复盘时
每周/每月回顾一次。

## 🗑️ 什么时候清理？

### 1. 过时的记录
如果某条规则已经固化到配置文件，可以删除。

**示例：**
```
MEMORY.md: "2026-03-06 ERROR 任务没有验收标准"
↓ 验证有效后
SOUL.md: "任务必须具备 owner、deadline、acceptance"
↓ 固化后
删除 MEMORY.md 中的记录
```

### 2. 重复的记录
如果多条记录说的是同一件事，合并成一条。

### 3. 无用的记录
如果某条记录从来没用过，可以删除。

## 💡 常见问题

### Q: MEMORY.md 和 SOUL.md 有什么区别？

**SOUL.md：**
- 价值观，定义了 AI 的行为准则
- 不常变
- 示例："聪明大于听话"

**MEMORY.md：**
- 记忆，记录了具体的经验教训
- 经常变
- 示例："2026-03-06 ERROR 任务没有验收标准"

---

### Q: 什么时候把 MEMORY.md 的内容写入 SOUL.md？

当某条经验教训被验证有效，并且需要成为"行为准则"时。

**流程：**
```
1. 在 MEMORY.md 记录
2. 验证有效（多次使用）
3. 固化到 SOUL.md
4. 删除 MEMORY.md 中的记录
```

---

### Q: MEMORY.md 会不会越来越长？

会的。建议：
1. 每月清理一次
2. 把重要的记录固化到 SOUL.md 或 AGENTS.md
3. 把详细的记录归档到 `memory/archive/` 目录

---

### Q: 可以不用 MEMORY.md 吗？

可以，但不推荐。

**没有 MEMORY.md：**
- AI 会重复犯错
- 无法积累经验
- 每次都要重新解释

**有了 MEMORY.md：**
- AI 自动避免重复错误
- 持续进化，越来越聪明
